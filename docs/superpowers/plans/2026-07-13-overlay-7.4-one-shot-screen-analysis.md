# Overlay 7.4 - One-shot Screen Analysis Implementation Plan

> Реализуется поверх завершенного Visual Capture (`VisualAsset` + `stageVisualAsset`), без второго image API и без второго voice runtime.

**Goal:** По явной голосовой просьбе пользователь подтверждает системный Android MediaProjection dialog, Vass захватывает один кадр без собственного overlay, привязывает его к исходной реплике и отвечает в существующем voice turn.

**Non-negotiable boundary:** голос запускает только системный dialog. Сам захват возможен исключительно после Android consent для каждой новой session. Один `MediaProjection` token и один `createVirtualDisplay()` используются ровно один раз.

## Core Flow

```text
voice: "объясни, что сейчас на экране"
  -> POST /chat/send uploads/transcribes audio as today
  -> server recognizes screen request before it creates Message/model call
  -> SSE: transcription + screenCaptureRequested, then [DONE]
  -> mobile hides overlay and opens ScreenCapturePermissionActivity
  -> Android system consent
  -> VassMediaProjectionService captures second valid frame, JPEG -> cache
  -> service stops projection/resources and restores overlay
  -> JS stages cache JPEG through existing Visual Asset API
  -> existing /chat/send with transcribed text + visualAssetId
  -> normal multimodal Gemini/TTS/history pipeline
```

The preflight is deliberately not `ExternalActionService`: that path runs after the primary model answer and would make the screenshot arrive one turn too late.

## Scope

- Android API 26+ one-shot capture, all compatible phone displays.
- Android 14+ system choice dialog (entire display/app window) via normal MediaProjection API.
- One JPEG, max long side 2048 px, quality reduced only as needed to remain below Visual Capture's 10 MiB contract.
- Text voice request is preserved; no separate OCR-only behavior.

Out of scope: continuous capture, video, background polling, AccessibilityService, screen thumbnails in logs, capture after consent cancellation, automatic retries, image memory and iOS.

## Step 1 - Server Preflight Contract

Files:
- Modify `VoiceAssistant.API/Controllers/ChatController.cs`
- Create `VoiceAssistant.API/Services/ScreenAnalysisIntentService.cs`
- Modify/add API unit and integration tests

1. Extend `SendRequest` with `SupportsScreenAnalysis` (false by default).
2. After audio transcription and before saving a `Message`, run an inexpensive candidate check for explicit screen vocabulary; only candidates use the small Gemini JSON classifier.
3. If classified `screen_analyze`, emit transcription plus `{ screenCapture: { prompt } }` SSE and terminal `[DONE]`; do not save a message, call reminders/memory/external actions or invoke the main model.
4. Delete the just-uploaded transient audio file on this branch; the retry is text + image and has no audio-history reference.
5. Skip preflight when `VisualAssetId` is already present: the model receives that user-selected visual immediately.
6. Classifier returns `chat` on malformed JSON/API failure. It never sees image bytes or private history beyond the minimal current text.

## Step 2 - Native Capture Module

Files:
- Create `ScreenCapturePermissionActivity.kt`, `VassMediaProjectionService.kt`, `ScreenCaptureStore.kt`, notification helper
- Modify `VassOverlayModule.kt`, `VassOverlayService.kt`, `OverlayContract.kt`, Expo plugin and TypeScript bridge

1. Add a distinct non-exported `mediaProjection` foreground service and `FOREGROUND_SERVICE_MEDIA_PROJECTION` declaration. It is never combined with the long-lived `specialUse` overlay service.
2. `requestScreenCapture(requestId)` hides overlay before launching an explicit permission Activity. If Android refuses a background start, persist `awaitingConsent` and expose a notification action that opens the same Activity on a real user tap.
3. Permission Activity calls `createScreenCaptureIntent()`. On success it starts the media-projection FGS first, then hands the one-time result token to it and finishes. Cancellation returns a typed `cancelled` result.
4. Projection service registers `MediaProjection.Callback` before display creation, builds one `ImageReader`, discards the first callback, encodes the second valid image as a bounded JPEG in `cacheDir/vass-screen-captures/`, then releases reader/surface/display/projection in one idempotent `finally` path.
5. A 4-second capture timeout, `onStop`, lock-screen/permission errors and black/empty result all clean resources and report one typed failure. No retry loop.
6. Persist only request/result metadata and cache URI long enough for JS recovery after a background event; no pixels/base64/preferences. Delete file after upload, cancellation or failed staging.
7. Overlay visibility is restored exactly once once native capture finishes. It must be `gone` through the complete capture window, not merely transparent.

## Step 3 - Conversation Runtime Handoff

Files:
- Modify `mobile/src/api/client.ts`, `mobile/src/hooks/useVoiceChat.ts`, `mobile/src/context/ConversationRuntimeContext.tsx`
- Create `mobile/src/screenCapture/*`

1. Add `screenCapture` SSE parsing and a typed callback that records the transcribed prompt; do not queue it alongside post-answer external actions.
2. After preflight `[DONE]`, pause recorders/TTS, request native capture and retain the turn generation token.
3. Receive/recover the cache URI, call existing `stageVisualAsset`, then send the preserved transcription through existing `/chat/send` with `visualAssetId`.
4. Consume the pending visual only after the second, actual multimodal turn reaches `[DONE]`. Consent cancellation/upload failure leaves no server attachment and returns listening to idle with a concise notice.
5. Prevent continuation/shadow recording from creating a second capture request while consent/capture/upload is active. Barge-in cancels only before the system dialog; after consent Android owns cancellation.
6. Advertise capability only when the native module exists on Android. Other platforms retain normal chat behavior.

## Step 4 - UI, Privacy And Failure States

Files:
- Modify overlay notification and Settings/Profile disclosure surface
- Modify `docs/react-native/BACKLOG.md` only when completed

1. Add concise Russian disclosure before first screen-analysis request: screenshot is sent only after the Android dialog and then used for the requested answer.
2. The capture notification says that screen analysis is in progress and exposes Cancel/Open Vass.
3. Cancellation is quiet; protected/black frame has a specific user message recommending normal screenshot/file attachment.
4. Do not show captured screenshot in the overlay itself. Existing pending preview/history behavior is enough once the file reaches Visual Capture.

## Verification Gates

- Backend intent service unit tests: direct Russian phrasing, non-screen request, malformed classifier response, no action with existing image.
- Integration test: preflight emits transcription + screen event, persists no message/attachment and removes transient audio; normal retry creates one visual attachment.
- Kotlin/JVM or focused native test for cleanup/state helper where feasible; `./gradlew :app:assembleRelease` is mandatory.
- `npx tsc --noEmit`, Expo config and backend test suites.
- Physical Android: consent deny, full-screen capture, app-window capture Android 14+, protected-screen fallback, rotation, overlay hidden in captured frame, return to conversation, upload/network failure and stale process recovery.
- Deploy API only after automated gates; release APK only after native build and signing verification.

## Rollback

The API capability flag defaults false and the client sends it only from the native Android build. If MediaProjection has an OEM-specific regression, disabling this capability restores the ordinary voice flow; Visual Capture and overlay remain unaffected.

## Sources

- Android requires a fresh user consent/token for every MediaProjection session and allows one `createVirtualDisplay()` per projection: <https://developer.android.com/media/grow/media-projection>
- A media-projection foreground service needs its own manifest type/permission and must start after consent: <https://developer.android.com/develop/background-work/services/fgs/service-types>
