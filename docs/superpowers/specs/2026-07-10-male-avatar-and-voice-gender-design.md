# Второй аватар (мужской) + разметка голосов по полу — design

## Контекст

Пользователь подготовил полный набор ассетов для второго, мужского аватара — `docs/designs/male_avatar_asset_plan.md` фиксирует источник и состав слоёв, файлы уже лежат в `mobile/assets/avatar/male_*.png`. Задача: дать пользователю переключаться между аватарами (Ольга/мужской), и при переключении — автоматически подставлять голос нужного пола. `expo-speech`'s `Voice` тип (`node_modules/expo-speech/build/Speech.types.d.ts`) не содержит поля пола (`{identifier, name, quality, language}` — всё), подтверждено кодом, не только устным замечанием в старом комментарии — значит определение пола голоса программно невозможно, только вручную.

Согласовано в брейншторме: ручная разметка голосов (не эвристика по имени, не отдельная память голоса на аватар) — единственный надёжный способ при этом ограничении платформы.

## Архитектура

### `LayeredAvatar` — переименование `OlgaLayeredAvatar.tsx`

Один параметризованный компонент вместо дублирования — оба аватара используют идентичную анимацию (моргание, цикл рта, halo, sleeping/paused-затемнение через `opacity`), различаются только набором PNG-слоёв.

```ts
export type AvatarId = 'olga' | 'male';

interface AvatarAssetSet {
  base: number;
  eyesClosedOverlay: number;
  mouthOpenSmallOverlay: number;
  mouthOpenBigOverlay: number;
  browsThinkingOverlay?: number; // у мужского нет — см. asset-план, thinking только меняет цвет halo
}
```

`AVATAR_ASSETS: Record<AvatarId, AvatarAssetSet>` — статическая карта с `require(...)` на существующие файлы (`olga_*`/`male_*`). `LayeredAvatarProps` получает новый обязательный `avatarId: AvatarId` рядом с уже существующими `state`/`sleeping`/`disabled`/`onLoadError`. Рендер brows-оверлея в `thinking` — условный: `{state === 'thinking' && assets.browsThinkingOverlay && <Image source={assets.browsThinkingOverlay} .../>}`.

`AvatarFace.tsx` (fallback при ошибке загрузки) не меняется — уже абстрактный, не привязан к конкретному аватару.

### Хранение выбора аватара — бэкенд, как `assistantName`

`AvatarId` — свойство аккаунта, не устройства (в отличие от голоса, который зависит от того, что установлено на конкретном телефоне). Синхронизируется между устройствами через device-link, как имя ассистента.

- `VoiceAssistant.API/Data/Entities/UserSettings.cs`: `public string? AvatarId { get; set; }` — nullable, `varchar(20)` (сопоставимо с `ClientLogEntries.Category`, фиксированный небольшой словарь значений). Новая EF-миграция.
- `SettingsController.cs`: `AvatarId` добавляется в `SettingsResponse`/`SettingsUpdateRequest`, без валидации набора значений на уровне бэкенда (как `AssistantName` — источник допустимых значений это мобильный клиент, не БД-констрейнт; новый аватар в будущем не потребует миграции схемы).
- `mobile/src/api/client.ts`: `Settings.avatarId: string | null`; новый `api.updateAvatarId(avatarId: string)`, зеркалит `updateNames` — round-trip всего объекта настроек (та же защита от случайного стирания `customSystemPrompt`/других полей, что уже документирована в комментарии над `updateNames`).
- `AuthContext.tsx`: `avatarId`/выставляется в том же эффекте, что уже грузит `assistantName`; сбрасывается в `null` при logout.
- Резолюция дефолта — на клиенте, не на бэкенде: `const displayAvatarId: AvatarId = avatarId === 'male' ? 'male' : 'olga';` — существующие пользователи без сохранённого значения (до этой фичи) получают `'olga'`, без миграции данных.

## UI: переключение аватара

Новая секция в `ProfileScreen.tsx` (только `mode === 'settings'`, как device-link/logout), между «Как назвать ассистента?» и «Голос ассистента» — заголовок «Аватар ассистента», два превью-портрета (по 72px, круглые, `require('.../olga_base.png')` / `require('.../male_base.png')`) рядом, выбранный подсвечен рамкой в акцентном цвете экрана (`#4a6fa5`, существующая палитра `ProfileScreen` — этот экран остаётся в светлой теме, AMOLED его пока не касался, см. BACKLOG «Фаза 4»).

Тап — немедленное действие, без отдельной кнопки «Сохранить» (как уже работает выбор голоса): `api.updateAvatarId(id)` → `refreshProfile()` → если среди уже загруженных голосов есть хотя бы один с тегом нужного пола, встаёт первый по порядку списка (тот же `setVoicePreference` + `previewVoice`, что уже используется при ручном выборе голоса, — пользователь СЛЫШИТ, что голос сменился). Если размеченных голосов нужного пола нет — голос не трогается.

`HomeScreen.tsx`: `assistantName || (displayAvatarId === 'male' ? 'Максим' : 'Ольга')`; `<OlgaLayeredAvatar .../>` → `<LayeredAvatar avatarId={displayAvatarId} .../>`.

## UI: разметка и группировка голосов

Хранение тегов — локально на устройстве (`SecureStore`, тот же паттерн, что `setVoicePreference`), не на бэкенде: голоса — свойство конкретного телефона, тег для NETWORK-голоса на одном устройстве не обязан существовать на другом.

`systemSpeech.ts`: `type VoiceGender = 'male' | 'female'`; `getVoiceGenderTags(): Promise<Record<string, VoiceGender>>` (с тем же in-memory кэшем, что `cachedRussianVoice`), `setVoiceGender(identifier, gender): Promise<void>`. Вычисление «первый голос нужного пола» — не отдельная функция в `systemSpeech.ts`, а прямо в `ProfileScreen.tsx` (там уже загружен и `voices`, и теги — пересчитывать через ещё один async-round-trip незачем).

Список голосов в `ProfileScreen.tsx` делится на три группы: **«Мужские голоса»**, **«Женские голоса»**, **«Не размечено»**. Пустая группа «Мужские»/«Женские» не рендерится вовсе; «Не размечено» рендерится, только если есть хоть один нетегированный голос — не «пустая с подсказкой» для этого случая, подсказка нужна только при попытке переключить аватар без единого размеченного голоса подходящего пола (см. выше). Нумерация «Голос N» — своя в пределах каждой группы, не сквозная по всему списку.

Каждая строка голоса получает маленький тумблер «М» / «Ж» (два тап-переключателя, активный подсвечен) — тегирование не зависит от прослушивания/выбора, можно разметить голос, даже если он сейчас не выбран. Существующая логика предпочтения локальных голосов (`isLikelyLocalVoice`, сеть-иконка 🌐) не меняется — группировка по полу накладывается поверх уже отфильтрованного `displayVoices`, а не заменяет его.

Осознанная граница: перетегирование УЖЕ выбранного голоса не форсирует его снятие и не триггерит автопереключение — авто-подбор происходит только в момент переключения аватара, не в момент правки тега.

## Обработка ошибок

Как сейчас: ошибка загрузки любого слоя `LayeredAvatar` (для любого `avatarId`) → `onLoadError` → `HomeScreen`'s `assetsFailed` → fallback на `AvatarFace` до конца сессии, без retry-петли — независимо от того, какой аватар был активен.

## Не в этой фиче

Автоопределение пола голоса (недоступно на платформе, см. Контекст). Более двух аватаров (архитектура `Record<AvatarId, ...>` это не блокирует, но экран выбора рассчитан на ровно 2 карточки). Личные/загружаемые аватары. Синхронизация тегов голосов между устройствами (осознанно локально, см. выше).

## Проверка

- `npx tsc --noEmit` после каждого таска.
- Веб-превью: переключение аватара (тап по превью → смена `avatarId`, смена дефолтного имени, смена голоса при наличии тега), группировка/тегирование голосов в реальном времени.
- Бэкенд: `AvatarId` round-trip через `GET`/`PUT /settings` зеркалит уже проверенный `AssistantName` — низкий риск, отдельного интеграционного теста не требует.
- Физический тест (как всегда для голосового цикла и реального рендера): читаемость мужского портрета на `AVATAR_SIZE=320`, отличимость `thinking` без смены бровей (только halo), тегирование голосов на реальном списке устройства.
