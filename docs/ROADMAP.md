# Дорожная карта — согласованные приоритеты

Живой документ. Фиксирует порядок работ, согласованный в обсуждении
2026-07-12, после закрытия всей программы `PROJECT-AUDIT-2026-07-10` и
добавления двух новых спек (visual capture, Android overlay). Не
конкурирует с [`react-native/BACKLOG.md`](react-native/BACKLOG.md) —
BACKLOG.md остаётся источником истины по конкретным таскам внутри
мобильных фаз, этот документ задаёт порядок МЕЖДУ фазами/спеками/новыми
направлениями (включая то, что не является мобильной фичой, например
админ-панель).

## Порядок

### 1. Админ-панель (новое направление, backend + отдельный web-фронт)

Список всех зарегистрированных пользователей с активностью (`LastActiveAt`,
`CreatedAt` — уже есть в `User`), количеством сообщений и объёмом символов
(агрегация по `Message.Content.Length`). Возможность подтверждать
регистрацию — инфраструктура нужна сейчас, реальный гейт (все продолжают
попадать автоматом) — нет. Задел под будущие лимиты токенов/моделей на
время бета-тестирования.

Технически дёшево: `AddIdentity<User, IdentityRole>` уже сконфигурирован
(роли не используются, но инфраструктура есть), нужна одна миграция
(`User.IsApproved`, default `true`), новый `AdminController`
(`[Authorize(Roles="Admin")]`). Старый `frontend/` полностью удалён в
ARCH-01 — новый web-фронт строится с нуля, React + Vite SPA (решено
2026-07-12, вместо статического HTML или экрана в мобильном приложении —
админ-задачи по таблицам данных плохо ложатся на телефон, а стандартный
SPA-стек проще растить фильтрами/графиками/пагинацией по мере усложнения
задач).

### 2. Долгосрочная память + напоминания (BACKLOG.md — Фаза 4, остаток)

Следующий пункт по уже существовавшему до новых спек плану (среднесрочная
память сдана — PR #69, далее долгосрочная). pgvector + `MemoryFacts` +
RAG-подмешивание. Напоминания идут в этом же пункте: технически это тот же
факт с датой-триггером, инфраструктура хранения/извлечения общая.

### 3. Android overlay — ядро (Фаза 7.1 + 7.2 + возврат в Vass без интентов)

Перенесено раньше по сравнению с исходным «после пилота» — осознанное
решение 2026-07-12: плавающий аватар должен появиться до того, как вокруг
него нарастает контент-функциональность (Intent-роутер, YouTube), чтобы
«открытие приложений» и overlay работали вместе с самого начала, а не как
два разрозненных релиза.

Технический нюанс, найденный при пересборке порядка: сама спека overlay
завязывает YouTube-часть (7.3) на Intent-роутер Фазы 5, а screen analysis
(7.4) — на visual capture. Оба этих под-этапа технически не могут
собраться раньше своих зависимостей. Решение: ядро overlay (7.1
foundation, 7.2 общий `ConversationRuntime`, «вернись в Vass» БЕЗ
голосовой маршрутизации) — самодостаточно и идёт в этом пункте. YouTube
через overlay (остаток 7.3) и screen analysis (7.4) достраиваются позже,
как только рядом появляются их зависимости (пункты 4 и 5 ниже) — без
разрыва по времени между «overlay существует» и «overlay умеет то, ради
чего его хотели».

### 4. Intent-роутер + YouTube (BACKLOG.md — Фаза 5, минимальный срез)

Пристыковывается к уже существующему из пункта 3 overlay — сразу даёт
полный сценарий «скажи Vass открыть видео, оно всплывает поверх». Остаток
контент-компаньона (новости/сериалы/рецепты) не блокирует overlay и может
идти следом или параллельно по мере appetite, отдельного приоритета не
имеет.

### 5. Visual capture (новая спека — фотофиксация)

Самодостаточна, не требует нативного Android-кода, работает как обычный
разговорный ход. Раскрывает screen analysis (7.4) у overlay постфактум —
после этого пункта overlay-функциональность становится полной.

### 6. Пилот (BACKLOG.md — Фаза 6)

Онбординг, метрики, решение об EAS/сторах. Сама overlay-спека требует
«решения о распространении» и результатов пилота перед hardening-этапом
ниже — так что это по-прежнему естественная точка перед финальным этапом.

### 7. Overlay hardening + distribution gate (Фаза 7.5)

Физическая матрица устройств, privacy/store compliance, Google Play
review либо managed sideload. Остаётся последним осознанно — завязано на
решение о распространении из пункта 6.

## Не в основной последовательности (opportunistic, без изменений)

- Нативный Android-модуль микрофона — только если реальный тест покажет
  проблему (эхо/задержка).
- Фоновый режим, кнопки гарнитуры, режим «для пожилых» — Фаза 2, остаток.
- Rive-аватар, рот по амплитуде TTS, эмоции по тегу LLM — Фаза 3, optional.
