# План: Voice Companion — нативное приложение (React Native)

> Целевой продукт (из презентации): голосовой ИИ-спутник для пожилых людей.
> Voice-first, без экранов и сложных интерфейсов, общение через беспроводные
> наушники. Тёплый собеседник + бытовой помощник + долговременная память.
> Этот план — «Этап 2: Мобильный додаток (RN)» из презентации, расширенный
> требованиями: встроенный нейро-TTS, анимированный аватар, трёхуровневая
> память, YouTube, новости, поиск сериалов.

## 0. Что уже доказано веб-прототипом (Этап 1)

Работает и переносится концептуально в RN:
- Полный голосовой цикл: VAD по RMS → запись → Gemini-транскрипция → LLM
  (Gemini 3.5 Flash + Google Search grounding) → стриминговый TTS (Piper, raw
  PCM) с проигрыванием по предложениям.
- Перебивание ассистента голосом, «дослушивание» продолжения фразы (shadow
  capture), фразы-заполнители при долгих операциях (preamble race).
- Голосовое управление поведением («запомни, говори медленнее») →
  постоянные инструкции в системном промпте.
- Ограничение краткосрочной памяти окном ~6000 токенов.
- Экономика подтверждена: self-hosted TTS снял основную статью расходов,
  Gemini Flash даёт целевую себестоимость ~$1/мес на пользователя.

Больные места веба, которые обязано закрыть нативное приложение:
- iOS Safari аудио-хаки (разблокировка жестом, AudioContext interrupted,
  глушение микрофона при воспроизведении → не работает перебивание на iPad).
- Нет работы в фоне/при заблокированном экране.
- Нет интеграции с кнопками гарнитуры.
- Задержка сервера на TTS (сеть) вместо мгновенной локальной озвучки.

## 1. Архитектурные решения

### 1.1 Каркас
- **React Native + Expo (dev client / prebuild)** — Expo-инструментарий
  (сборка, OTA-обновления через EAS Update — аналог нынешнего «git pull и
  готово»), но с нативными модулями, которые понадобятся для аудио.
- Один кодовый репозиторий с бэкендом (монорепо: `mobile/` рядом с
  `VoiceAssistant.API/`).
- TypeScript.

### 1.2 Аудио-стек (ядро продукта)
- **Аудио-сессия**: нативный модуль поверх `AVAudioSession` (iOS,
  `playAndRecord` + `voiceChat`-режим с эхоподавлением) и `AudioManager`
  (Android). Это снимает всю категорию iOS-хаков веба: full-duplex
  запись+воспроизведение штатно поддерживается ОС.
- **VAD on-device: Silero VAD через onnxruntime** (модель ~2МБ, создана
  именно для этого). Замена самодельного RMS-порога: различает речь и
  кухонный шум (требование презентации), устойчива к расстоянию до
  микрофона (проблема, из-за которой пришлось приостановить голосовую
  идентификацию).
- **Кнопки гарнитуры**: `MPRemoteCommandCenter` (iOS) / MediaSession
  (Android) — play/pause как push-to-talk / перебивание.
- **Фоновый режим**: iOS `audio` background mode (законный кейс — приложение
  непрерывно пишет/воспроизводит), Android — foreground service с
  уведомлением.

### 1.3 Встроенный нейро-TTS (требование: без сети, мгновенно)
- **Рекомендация: sherpa-onnx TTS** — рантайм с готовыми мобильными
  сборками (iOS/Android), напрямую исполняет **те же Piper-модели**
  (`ru_RU-irina-medium`, уже выбранный голос!) на устройстве через ONNX
  Runtime. Модель ~60МБ в бандле, синтез быстрее реального времени на
  телефонных чипах.
  - Плюс: голос идентичен текущему прод-голосу — переход незаметен.
  - Запасной вариант A: системный `AVSpeechSynthesizer` / Google TTS
    (нулевая интеграция, но качество русских голосов хуже и различается
    между устройствами).
  - Запасной вариант B: серверный Piper как fallback при проблемах на
    слабых устройствах (эндпоинт уже есть).
- Стриминг по предложениям сохраняем: клиент режет ответ LLM по
  пунктуации (логика из `yolo.js` переносится 1:1) и синтезирует локально
  каждое предложение — исчезает даже сетевой хоп до VPS.

### 1.4 STT (распознавание)
- Оставить **серверную Gemini-транскрипцию** (работает, качество
  подтверждено, дёшево). Аудио шлём как сейчас (webm/m4a upload).
- Опция на будущее (не в первых фазах): on-device Whisper
  (whisper.cpp small) для офлайн-режима — отдельное исследование.

### 1.5 Анимированный аватар
- **Рекомендация: Rive** (`rive-react-native`) — векторные state machines,
  крошечный рантайм, 60fps на слабых устройствах, дёшево в производстве.
  - Состояния: idle (моргание/дыхание) → listening (наклон, «внимание») →
    thinking (взгляд вверх) → speaking (рот открывается по амплитуде).
  - Синхронизация рта: RMS-огибающая из локального TTS-буфера (у нас же
    сырой PCM — амплитуда бесплатна) → input в Rive state machine.
    Полноценные виземы (фонемная артикуляция) — опционально позже, Piper
    отдаёт фонемные метки, их можно замапить на 4-6 форм рта.
  - Для пожилых: крупное дружелюбное лицо на весь экран = «есть с кем
    говорить», а не абстрактный шарик.
- Альтернатива (дороже): 3D-аватар (react-three-fiber + Ready Player Me).
  Не рекомендую в первые релизы: вес, батарея, сложность.

### 1.6 Память (краткосрочная / среднесрочная / долгосрочная)
Всё серверное (телефон — тонкий клиент), бэкенд уже наш:
- **Краткосрочная** (есть): скользящее окно последних сообщений
  (~6000 токенов) — сделано.
- **Среднесрочная** (новое): скользящее резюме сессии. Фоновая задача
  (по образцу выкинутого NightlyAnalysisJob, но на Gemini): раз в N
  сообщений сворачивает выпавшую из окна историю в компактное резюме,
  которое включается в системный промпт. Хранение: `ChatSession.Summary`.
- **Долгосрочная** (новое): RAG на **pgvector** (расширение к уже
  работающему Postgres):
  - Извлечение фактов: параллельный side-call Gemini (паттерн уже
    отработан трижды: preamble, имена, инструкции) — вылавливает атомарные
    факты («лекарства лежат в верхнем ящике», «внук Миша приезжает в
    субботу») из каждой реплики.
  - Таблица `MemoryFacts(id, userId, fact, embedding vector(768),
    createdAt, lastRecalledAt)`; эмбеддинги — Gemini `text-embedding-004`
    (дёшево, то же API).
  - На каждый ход: эмбеддинг реплики → `ORDER BY embedding <=> query
    LIMIT 5` → top-K фактов в системный промпт.
  - Голосовая команда «запомни ...» уже работает (CustomSystemPrompt) —
    остаётся для *поведенческих* предпочтений; *фактическая* память
    переезжает в RAG.

### 1.7 Контент-компаньон (YouTube, новости, сериалы)
Единый механизм — **intent-роутер** перед основным LLM-ответом
(расширение уже работающего preamble-паттерна): лёгкий Gemini-вызов
классифицирует реплику → `{intent: chat|youtube_play|news_read|
series_search|recipe_read, params}`.
- **YouTube**: `react-native-youtube-iframe` (официальный IFrame-плеер,
  соответствует ToS). Поиск — YouTube Data API v3 (бесплатная квота
  10k units/день с головой хватает). Голосом: «включи видео про …» →
  intent → поиск → автоплей первого результата → голосовое управление
  («стоп», «дальше», «громче»).
- **Новости**: Gemini + Search grounding уже умеет (доказано погодой) —
  для «почитай новости» добавляется intent, который просит LLM собрать
  сводку из свежих новостей и озвучивает её по предложениям. Опция:
  RSS-агрегатор на бэкенде для стабильных источников.
- **Сериалы/фильмы**: TMDB API (бесплатный, есть русская локализация,
  says «где посмотреть» через watch-providers) — intent `series_search`:
  «найди сериал про …» → карточка + голосовое описание.
- **Рецепты** (сценарий №1 презентации): режим пошагового чтения с
  паузами — intent `recipe_read`, состояние «шаг N», команды
  «дальше»/«повтори»/«стоп». Источник: Search grounding или
  Spoonacular/Edamam.

### 1.8 Бэкенд
- Остаётся текущий .NET API на VPS (chat, память, grounding, интенты).
  Мобильному клиенту нужны небольшие правки:
  - Версионирование API (`/api/v1/...`) перед первым релизом приложения.
  - Упрощённый вход для пожилых: device-token / магическая ссылка от
    родственника вместо email+пароль (настраивает член семьи).
  - Push-уведомления (напоминания из памяти: «вы просили напомнить…») —
    FCM/APNs, серверная джоба.
- Чистка хвостов Polish Tutor — по файлу аудита (см. AUDIT-LEGACY.md).

## 3. Риски и открытые вопросы

| Риск | Митигация |
|---|---|
| iOS фоновый доступ к микрофону — App Review строг к постоянному слушанию | Честный audio-режим (мы реально играем/пишем звук), кнопка гарнитуры как основной триггер, описать кейс в review notes |
| sherpa-onnx TTS на старых Android — скорость | Fallback на серверный Piper (эндпоинт остаётся) |
| Silero VAD в шумной кухне | Порог + минимальная длительность речи уже отработаны на вебе; VAD-модель обучена на шуме |
| YouTube ToS (фоновое проигрывание запрещено в IFrame) | Видео — только с открытым экраном; в фоне только собственный TTS/контент |
| Стоимость side-call'ов (preamble, intent, факты — 3 вызова/реплика) | Все на Flash-модели без grounding (~копейки); при росте объединить в один структурированный вызов |
| Пожилой пользователь молчит → приложение «умирает» | Проактивные реплики по таймеру/расписанию («как прошла прогулка?») — фаза 4+, использует память |

## 4. Что сознательно НЕ делаем в первых релизах
- 3D-аватар, липсинк по виземам (только амплитуда).
- On-device STT/LLM (офлайн-режим) — отдельный трек после пилота.
- Идентификация говорящего по голосу — приостановлена (см. коммит
  0ecb2b9), вернёмся при появлении спроса от пилота; инфраструктура
  (speaker-id сервис, таблица, кластеризация) сохранена.
- Мультиязычность — только русский до пилота.
