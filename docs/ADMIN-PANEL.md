# Админ-панель Vass

Статус: `implemented` (2026-07-12).

Панель собрана отдельным React + Vite SPA в `admin/` и публикуется через
общий nginx по адресу `/admin/`. Она использует существующий JWT-вход Vass;
доступ к admin API разрешён только пользователям с ролью `Admin`.

## Что реализовано

- сводка: пользователи, активные за 24 часа и 7 дней, сообщения, символы;
- таблица с email, датой регистрации, последней активностью, количеством
  сообщений и суммой `Message.Content.Length`;
- поиск, фильтр статуса, сортировка и серверная пагинация;
- подтверждение и ограничение доступа пользователя;
- защита администратора от блокировки собственной учётной записи;
- немедленная инвалидация JWT пользователя при ограничении доступа;
- автоматическое подтверждение новых регистраций, пока идёт открытая бета.

Индикатор «сейчас активен» в UI означает активность в последние пять минут,
а не постоянное WebSocket-присутствие.

## Первый администратор

1. Зарегистрировать обычную учётную запись Vass.
2. Указать её email в `ADMIN_EMAIL` (`Admin__Email` без Docker).
3. Перезапустить API. При старте `AdminBootstrapper` создаст роль `Admin`,
   назначит её найденной учётной записи и обновит security stamp.
4. Войти заново на `/admin/`: ранее выданный JWT после назначения роли
   намеренно недействителен.

Если учётная запись с указанным email ещё не создана, API только запишет
предупреждение в лог. Пароль администратора через конфигурацию не хранится.

## Локальный запуск

```powershell
dotnet run --project VoiceAssistant.API
cd admin
npm ci
npm run dev
```

Панель: `http://127.0.0.1:4173/admin/`. Dev proxy отправляет `/api` на
`http://localhost:5000`; другой адрес задаётся переменной
`VITE_DEV_API_TARGET`.

Production-сборка и проверки:

```powershell
cd admin
npm run typecheck
npm run build
npm audit --audit-level=high
```

В Docker панель собирается отдельным сервисом `admin`; внешний путь остаётся
`/admin/`. `scripts/deploy.sh` пересобирает `api` и `admin`, перезапускает
nginx и проверяет оба endpoint после выкладки.

## Конфигурация доступа

```env
REGISTRATION_AUTO_APPROVE=true
ADMIN_EMAIL=admin@example.com
```

При `REGISTRATION_AUTO_APPROVE=false` регистрация создаёт пользователя со
статусом ожидания и возвращает HTTP 202 без JWT. До включения этого режима в
production мобильному клиенту нужен отдельный экран «ожидает подтверждения»;
при текущем `true` его существующий сценарий не меняется.

## Admin API

| Метод | Endpoint | Назначение |
|---|---|---|
| `GET` | `/api/v1/admin/overview` | Сводные показатели |
| `GET` | `/api/v1/admin/users` | Поиск, фильтры, сортировка, пагинация |
| `PATCH` | `/api/v1/admin/users/{id}/approval` | Подтвердить или ограничить доступ |

## Следующий инкремент

Точные токены, используемые модели и стоимость пока не сохраняются, поэтому
панель не подменяет их оценками по длине текста. Для контроля beta-бюджета
нужна отдельная сущность AI usage на каждый вызов: пользователь, провайдер,
модель, input/output tokens, endpoint, latency, status и стоимость по
действовавшему тарифу. После этого в панели появятся срезы и лимиты без
искажения исторических данных.
